바탕화면에 이미지를 저장할 폴더 만들기

	사진을 저장하고 불러올 때, SQL 칼럼 자료형으로 
	varchar2 를 사용할 수도 있고, 이미지 자체를 저장하는 blob 을 사용할 수도 있다.
	하지만 DB 에 이미지 자체를 저장하게 되면, 이미지가 훼손될 가능성이 있다.
	이미지를 글자로 변환할 때, 변환 과정에서 문제가 생길 가능성이 크다.
	또한, DB 에 이미지 자체를 저장하게 되면, 비용이 많이 들기 때문에 많이 사용하지 않는 방법이다.
	따라서, 이미지를 저장할 때는 SQL 자료형 varchar2 로 이미지의 경로를 설정해주고, 
	설정한 이미지의 경로를 가져오는 방식을 주로 사용한다.
	
	이미지가 저장되는 경로
	src/main/resources
	             └ static   ◀ css, js, font 와 같이, 공개가 가능하고 유출되어도 상관없는 코드를 여기에 넣어준다.
	                  └ img ◀ 홈페이지에서 공개해도 상관없고, 공통으로 사용할 수 있는 이미지를 넣어준다.
	관리자가 지정한 이미지 저장 경로 ◀ 관리자가 아닌 사용자가 업로드하거나, 관리자가 비공개로 이미지를 가져오고 저장하는 경로
	
	라이브러리나 프레임워크가 설정되지 않은 js 코드는 유출될 가능성이 높다.
	라이브러리나 프레임워크로 설정된 js 코드는, 
	코드의 형식이 라이브러리나 프레임워크의 형식으로 작성되어 있기 때문에,
	외부에서 코드를 유추하기 어렵다.
	
	만약, 이미지를 저장하고 싶은 폴더가 존재하지 않거나, 의도치 않게 사라지거나,
	폴더 안에 이미 존재하는 이미지의 이름과 형식이 모두 일치하여 기존의 이미지를 덮어 씌우거나,
	이미지가 제대로 저장되지 않을 가능성이 있기 때문에,
	
	이미지를 저장할 때는
	
	1. 이미지를 불러오거나 저장할 경로가 확실히 존재하는지 확인하기
		▶ 해당하는 경로 또는 이미지가 존재하지 않아 불러오지 못할 때: 이미지를 불러오는데 실패했습니다. 라는 이미지 띄워주기
		▶ 이미지를 저장할 때, 저장할 폴더가 존재하지 않을 경우: 이미지 경로 확인 후, 원하는 경로가 없다면, 이미지를 저장하기 위해 경로 생성해주기
	
	2. 이름이 중복될 경우
		▶ 이미지를 저장할 때, 이미지의 이름이 중복될 경우: 이름과 형식이 동일한 이미지가 이미 존재하여, 이미지가 덮어 씌워지거나, 저장되지 않을 경우,
		                                      이름의 중복을 피하기 위해 저장 시간을 이름 뒤에 붙여 저장한다.
		                                      ex) 가방.jpg
		                                          가방시분초.jpg
